# LANGUAGE — Pickle Rick Grok Architecture & Engineering Glossary

**Purpose**: Shared, precise vocabulary for all architecture, design, and improvement work. Use these terms *exactly*. Consistent language is the whole point — it makes the codebase (and future agent runs) dramatically more navigable and lower-entropy.

Do not substitute generic words like "component", "service", "API", or "boundary" when one of the terms below is more precise.

## Core Terms

**Module**  
Anything that has an **Interface** and an **Implementation**. Scale-agnostic: applies equally to a function, class, package, file, or cross-cutting slice of the system.

**Interface**  
Everything a caller (or test) must know to use the module correctly. Includes the type signature *plus* invariants, ordering constraints, error modes, required configuration, performance characteristics, and side-effect contracts.  
The surface that produces **Leverage** and **Locality**.

**Implementation**  
The actual code inside the module that delivers the behavior promised by the Interface. Distinct from **Adapter** (see below).

**Depth**  
The amount of useful behavior hidden behind a small, stable Interface.  
- A **Deep Module** has a simple Interface that hides a large, powerful Implementation (high leverage).  
- A **Shallow Module** has an Interface that is nearly as complex as its Implementation (low leverage, leaky, expensive to learn and maintain).

Depth is a property of the *Interface*, not the Implementation.

**Seam** (from Michael Feathers)  
A place where behavior can be altered *without editing the module itself*. The location at which a Module's Interface lives.  
- **Internal Seam**: Private to the module's own Implementation (used by its tests).  
- **External Seam**: Exposed to callers outside the module.

**Adapter**  
A concrete thing that satisfies an Interface at a Seam. Describes the *role* it fills, not its internal substance.  
One adapter = hypothetical seam. Two adapters = real seam (behavior actually varies).

**Leverage**  
What *callers* (and tests) gain from Depth: more capability per unit of Interface they have to learn. One deep Implementation can pay back across many call sites.

**Locality**  
What *maintainers* gain from Depth: change, bugs, knowledge, and verification are concentrated in one place rather than spread across callers. "Fix once, fixed everywhere."

## Key Principles

- **The Deletion Test**: Imagine deleting the Module entirely. If the surrounding complexity disappears, it was a pass-through (shallow). If the complexity reappears across many callers, the Module was earning its keep (deep).
- **The Interface is the Test Surface**: Callers and tests cross the same Seam. If you have to reach inside the Implementation to test meaningfully, the Interface is the wrong shape.
- **Depth Produces Leverage + Locality**: These are the two measurable payoffs.
- **Respect Prior Decisions**: Architectural choices recorded in ADRs or CONTEXT.md are load-bearing. Re-proposing something that contradicts them requires explicit justification and usually a new ADR.

## Supporting Concepts (PRG Extensions)

**Trap Door**  
A permanent, machine- and human-readable note left in the codebase or documentation (AGENTS.md, CLAUDE.md, etc.) that captures knowledge that must survive future agent runs. Created by convergence drivers (Anatomy Park, Szechuan, Citadel, future Deepener) when they surface something important.

**ConvergenceLoop**  
The reusable iteration engine (`engine/src/iteration.ts`) that drives measure → apply → gate → persist cycles with rollback on regression. Used by Anatomy Park, Szechuan, Microverse, and the future Architecture Deepener.

**ManagerRitual**  
The single source of truth for post-worker handling (`engine/src/ritual.ts`). Enforces promise tokens, artifact contracts, ConvergenceGate, CircuitBreaker, and precise rollback.

**grokRoot / Target Root**  
The deterministic root of the tree being edited (resolved by `SessionManager.getWorkingDirSafe`). Critical for self-dogfood so the machine always edits the correct repository.

**R-META Ticket**  
A self-improvement ticket (prefixed `R-META-`) generated by the self-PRD generator that targets improvements to the meta system itself (ritual, persist, citadel, architecture tooling, etc.).

## How to Use This Language

- When proposing a refactor, describe it in terms of **Deepening a Module at a specific Seam**, the resulting **Leverage** for callers and **Locality** for maintainers, and how the **Deletion Test** would change.
- When scanning for opportunities (Anatomy Park, future Deepener, self-PRD), classify findings using the above terms.
- When "grilling" a design (inside `/pickle-refine-prd` or a dedicated `/deepen` session), challenge proposals against the glossary and existing ADRs/CONTEXT.md.
